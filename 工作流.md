# 研电赛技术论文 — 协作工作流

本文档说明如何在本地编译本仓库中的 LaTeX 论文，以及如何与队友用 Git / GitHub 协作。

---

## 1. LaTeX 编译环境（请与维护者对齐）

维护者当前环境如下；**队友请尽量安装相同主版本**，以减少宏包与行为差异。


| 项目                   | 维护者本机                                                                 |
| -------------------- | --------------------------------------------------------------------- |
| 发行版                  | **TeX Live 2025**                                                     |
| 官方获取                 | [TeX Live](https://tug.org/texlive/)                                  |
| 工具链管理                | `tlmgr`（维护者：revision 74241；安装根目录 `D:/mysoftware/texlive/2025`）        |
| 必选引擎                 | **XeLaTeX**（`xelatex`）— 项目使用 `ctex`、`fontspec`，**不可用 pdfLaTeX 作为主流程** |
| XeTeX 版本参考           | `XeTeX 3.141592653-2.6-0.999997 (TeX Live 2025)`                      |
| 推荐辅助工具               | **latexmk**（与 `xelatex` 同在 TeX Live 的 `bin` 目录）                       |
| 维护者 `xelatex` 路径（示例） | `D:\mysoftware\texlive\2025\bin\windows\xelatex.exe`                  |


### 1.1 队友安装步骤（Windows / macOS / Linux）

1. 安装 **TeX Live 2025**（Windows 使用安装器 `install-tl-windows.exe`；macOS / Linux 使用 `install-tl`）。若只能安装更新年份的 TeX Live，编译报错时可用 `tlmgr install <包名>` 补齐缺失宏包。
2. 将 TeX Live 的 `bin` 目录加入系统 **PATH**（安装程序通常已配置；若终端找不到命令需手动添加）。
3. 在终端中校验：
  - `xelatex --version` — 输出中应含 `TeX Live 2025`（或你们约定的一致版本）。
  - `latexmk -v` — 确认已安装（推荐）。

### 1.2 字体

`technical_paper.tex` 中通过 `fontspec` 指定了 **Times New Roman、Arial、Consolas** 等。请确保本机已安装相应字体；若缺字，请根据 `.log` 报错补装字体，或在团队一致同意的前提下修改 `technical_paper.tex` 中的 `\setmainfont` / `\setsansfont` / `\setmonofont`。

### 1.3 编译命令

在项目根目录（与 `technical_paper.tex` 同级）执行：

**方式 A — 单次编译（目录/交叉引用不完整时，可连跑 2～3 次）：**

```bash
xelatex -interaction=nonstopmode technical_paper
```

**方式 B — 推荐（自动判断重复运行）：**

```bash
latexmk -xelatex -interaction=nonstopmode technical_paper
```

**本地连续预览（保存即编译）：**

```bash
latexmk -xelatex -pvc technical_paper
```

## 3. 图表、脚注与参考文献协作约定（按附件 3）

为避免多人协作时图号、表号、引用编号混乱，统一按以下规则写作。

### 3.1 图表编号

附件 3 要求图序、表序格式为 **「图 m-n」「表 m-n」**，并且 **每章重新编号**，图与表分开编号。

- 不要在正文中手写“图 1”“表 2”等固定数字。
- 正文引用图表时，统一使用 `\label` + `\ref`。
- 每个 `\label` 必须全局唯一，建议按章节和内容命名。
- 同一章多人协作时，先约定各自负责的图表范围，避免重复标签。

推荐命名规则：

```tex
fig:design_system_architecture
fig:hardware_power_circuit
fig:software_flow
tab:design_spec
tab:test_plan
```

正文引用模板：

```tex
系统总体架构如图 \ref{fig:design_system_architecture} 所示。
主要技术指标见表 \ref{tab:design_spec}。
```

图片模板：

```tex
\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.8\textwidth]{images/example.png}
  \caption{系统总体架构}
  \label{fig:design_system_architecture}

  {\zihao{5}\songti 资料来源：作者自绘。}
\end{figure}
```

表格模板：

```tex
\begin{table}[htbp]
  \centering
  \caption{系统主要技术指标}
  \label{tab:design_spec}
  \begin{tabular}{lll}
    \hline
    指标 & 目标值 & 说明 \\
    \hline
    输出电压 & 220 V AC & 示例 \\
    \hline
  \end{tabular}

  {\zihao{5}\songti 资料来源：作者整理。}
\end{table}
```

### 3.2 图表编号编译注意事项

图表编号、目录页码、交叉引用都由 LaTeX 在编译时自动生成。

- 新增、删除或移动图表后，必须至少连续编译 2 遍。
- 推荐使用 `latexmk -xelatex -interaction=nonstopmode technical_paper`。
- 若只用 `xelatex`，建议连跑 2～3 次。
- 定稿前检查 `.log` 中是否还有 `undefined references`、`multiply defined labels` 等提示。

### 3.3 脚注引用

附件 3 要求引用他人成果与数据处使用脚注，脚注全文连续编号，序号样式为 ①②③。

正文中引用他人成果、实验数据、标准、网页资料时，建议直接在对应句子后添加脚注：

```tex
MPPT 控制算法常采用扰动观察法、电导增量法等实现最大功率点跟踪\footnote{Hohm D P, Ropp M E. Comparative study of maximum power point tracking algorithms. Progress in Photovoltaics: Research and Applications, 2003, 11(1): 47--62.}。
```

网页资料脚注模板：

```tex
GD32G553 微控制器集成高性能处理器内核及多种外设资源\footnote{兆易创新. GD32G553 系列微控制器用户手册. 访问地址：TODO，访问日期：2026 年 TODO 月 TODO 日。}。
```

### 3.4 参考文献列表

当前项目的参考文献采用手写 `enumerate` 列表，编号由条目顺序自动生成。

- 不要在正文中手写 `[1]`、`[2]` 指代某篇文献，避免插入新文献后编号错乱。
- 新增参考文献时，优先追加到 `sections/07-references.tex`。
- 若必须调整参考文献顺序，应同时检查正文中是否有手写编号残留。
- 定稿前统一清理 `TODO`、补齐版本号、发布日期、访问日期等信息。

参考文献条目模板：

```tex
\item 作者. 题名. 期刊名, 年份, 卷(期): 起止页码.
```

书籍模板：

```tex
\item 作者. 书名. 版本. 出版地: 出版社, 年份.
```

技术手册模板：

```tex
\item 厂商. 芯片或产品系列用户手册. 版本号, 发布日期.
```

网页模板：

```tex
\item 机构或作者. 页面标题. 网址, 访问日期: YYYY-MM-DD.
```

### 3.5 定稿前检查清单

- 图题在图下方，表题在表上方。
- 图号、表号使用自动编号，不手写固定数字。
- 每个图表都有唯一 `\label`。
- 正文引用图表使用 `\ref`。
- 他人成果、数据、标准、网页来源已添加脚注或参考文献。
- 编译至少 2 遍后，无 `undefined references`。
- 无残留 `TODO`。

> 注意：附件 3 要求图表按章编号为「图 m-n」「表 m-n」。若当前编译结果仍为全文连续编号，需要在 `technical_paper.tex` 中补充按章重置图表编号的 LaTeX 配置。

### 4.2 HTTPS 推送 / 拉取

使用 HTTPS 时，GitHub 已不支持账户密码；请在提示密码处使用 **Personal Access Token（PAT）**（[GitHub 文档：创建 token](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/creating-a-personal-access-token)）。也可改用 SSH：将 `origin` 改为 `git@github.com:DAJIANGW/GD-paper.git` 并配置本机 SSH 密钥。

仓库维护者需在 GitHub：Settings → Collaborators 邀请协作成员。

## 5. 分支与提交习惯

- **小改、单章**：可直接在 `main` 上工作（团队规模小时），或自建短分支。
- **大段改写 / 新章**：建议 `feature/章节-简述`，合并前本地确保 `xelatex` / `latexmk` **无错误编译通过**。
- **提交说明**：建议加前缀便于浏览，例如：`feat:` 新内容、`fix:` 修正、`docs:` 文档/格式。

## 6. 协作分工建议

按章节文件认领，减少两人同时编辑同一 `.tex`：

- `sections/00-cover.tex`、`abstract.tex`
- `sections/01-innovation.tex` … `07-references.tex`

若必须同文件协作，先沟通段落边界；出现合并冲突时保留双方语义，统一术语与参考文献写法。

### 6.1 多人协作冲突规避

本模板按章节拆分成独立 `.tex` 文件，天然适合分工。**只要每人只改自己负责的章节文件，即使别人文件中对应章节还是空白，合并时也不会冲突。**

#### 不会冲突的情况

- 队友 A 修改 `sections/01-innovation.tex`，队友 B 修改 `sections/02-design.tex`。
- 两人各自在 `images/` 下新增图片，只要文件名不重复。
- 各章节内使用彼此独立的 `\label`。

#### 会冲突 / 会出错的情况

- **同时修改 `technical_paper.tex`**：例如都加宏包、改章节顺序、调格式。
- **`\label` 重名**：例如两章都给图起名叫 `fig:system`，交叉引用会乱。
- **图片文件名重复**：同名文件互相覆盖，最后合并时只保留一个。
- **改动公共文件未同步**：例如同时编辑 `sections/07-references.tex` 或 `abstract.tex`。

#### 推荐约定

1. **主文件专人维护**：版式、宏包、章节顺序由一人统一修改，其他人先在群里说明需求。
2. **标签加章节前缀**：
   - 图：`fig:software-flow`、`fig:hardware-circuit`
   - 表：`tab:design-spec`、`tab:test-plan`
   - 公式：`eq:mppt-algorithm`
3. **图片名加章节前缀**：例如 `fig_software_arch.png`、`fig_hardware_circuit.pdf`。
4. **图片统一放 `images/`**：不要散落在各章目录，方便管理和打包提交。
5. **提交前先 pull**：推送前执行 `git pull origin main`，减少远程冲突。
6. **不要删除别人的空白文件**：即使某章暂时空白，也不要删除其 `sections/*.tex`，否则主文件 `\input` 会报错。

## 7. 与远程已有提交冲突时

若 GitHub 上建库时勾选了 README / License，首次 `git push` 可能被拒绝。可先在本地执行：

```bash
git pull origin main --rebase
```

解决冲突后再 `git push -u origin main`（具体以当时分支名为准）。
